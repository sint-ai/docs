# Operators Platform

The SINT Operators Platform (`sint-agents`) is a production-grade enterprise dashboard for orchestrating AI agent fleets. It provides a unified control plane for agent lifecycle management, task dispatching, monitoring, compliance, and billing — built on React 19 and Redux Toolkit 2.5 with a feature-based modular architecture.

<Note>
  Repository: `sint-agents`. The Operators Platform is the primary interface for enterprise customers managing multi-agent deployments at scale.
</Note>

The platform is organized around operator workflows: onboard agents, define tasks, monitor execution, enforce security policies, and track costs. Every module is independently navigable and backed by its own Redux slice with typed state and async thunks.

## Architecture

### Frontend Stack

- **React 19**: Concurrent rendering, Suspense boundaries for async data, and Server Components where applicable.
- **Redux Toolkit 2.5**: All application state lives in Redux. Each feature module owns its slice — actions, reducers, selectors, and async thunks are colocated with the feature.
- **RTK Query**: API data fetching and caching layer. Auto-generates hooks from endpoint definitions, handles cache invalidation, and manages loading/error states.
- **React Router v7**: File-system-adjacent routing with 26 defined routes. Route-level code splitting via lazy imports.

## State Management

The store is configured in `src/store/index.ts` using `configureStore` from Redux Toolkit. All 30+ slices are registered at startup; RTK Query APIs inject their own reducers automatically.

### Slice Architecture

Each slice uses `createSlice` with `createAsyncThunk` for side effects:

```typescript
// Example: agentsSlice.ts (abbreviated)
import { createSlice, createAsyncThunk } from '@reduxjs/toolkit';

export const fetchAgents = createAsyncThunk(
  'agents/fetchAll',
  async (_, { rejectWithValue }) => {
    const res = await agentsApi.list();
    if (!res.ok) return rejectWithValue(await res.json());
    return res.json();
  }
);

const agentsSlice = createSlice({
  name: 'agents',
  initialState: { items: [], status: 'idle', error: null },
  reducers: {
    agentUpdated: (state, action) => { /* optimistic update */ },
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchAgents.pending, (state) => { state.status = 'loading'; })
      .addCase(fetchAgents.fulfilled, (state, action) => {
        state.status = 'succeeded';
        state.items = action.payload;
      })
      .addCase(fetchAgents.rejected, (state, action) => {
        state.status = 'failed';
        state.error = action.payload;
      });
  },
});
```

### Test Distribution

<Tabs>
  <Tab title="Slice Tests">
    Redux slices are tested in isolation. Each `createAsyncThunk` has tests for pending, fulfilled, and rejected paths. Reducers are tested against initial state and edge cases.

    ```typescript
    // agentsSlice.test.ts (example)
    describe('agentsSlice', () => {
      it('sets status to loading on fetchAgents.pending', () => {
        const state = agentsReducer(undefined, fetchAgents.pending('', undefined));
        expect(state.status).toBe('loading');
      });

      it('populates items on fetchAgents.fulfilled', () => {
        const agents = [{ id: '1', name: 'TestAgent' }];
        const state = agentsReducer(undefined, fetchAgents.fulfilled(agents, '', undefined));
        expect(state.items).toEqual(agents);
        expect(state.status).toBe('succeeded');
      });
    });
    ```
  </Tab>
  <Tab title="Component Tests">
    Feature components are tested with React Testing Library. Tests assert on rendered output and user interactions, not implementation details.

    ```typescript
    // AgentsPage.test.tsx (example)
    it('renders agent list after fetch', async () => {
      server.use(http.get('/api/agents', () => HttpResponse.json(mockAgents)));
      render(<AgentsPage />, { wrapper: StoreWrapper });
      await screen.findByText('TestAgent');
      expect(screen.getAllByRole('row')).toHaveLength(mockAgents.length + 1); // +1 header
    });
    ```
  </Tab>
  <Tab title="Integration Tests">
    Integration tests validate full feature flows: user dispatches an action → store updates → component re-renders correctly.

    ```bash
    # Run full test suite
    pnpm test

    # Run with coverage
    pnpm test --coverage

    # Run specific feature
    pnpm test src/features/agents
    ```
  </Tab>
</Tabs>

<Warning>
  Tests requiring WebSocket connections use MSW (Mock Service Worker) with WebSocket interceptors. Do not mock the Redux store directly in integration tests — use the real store with MSW-intercepted API calls for accurate coverage.
</Warning>
